              首次推送：
              
              
    1.git init：初始化本地 Git 仓库，在当前目录创建.git 隐藏文件夹，开始版本控制。
    2.git branch -m main ：将本地默认分支从master重命名为main（GitHub当前默认分支为main，避免分支名不一致）
    3.git add .：将当前目录下所有修改 / 新增的文件添加到 Git 暂存区（ staging area），. 代表所有文件。
    4.git commit -m "xxx"：将暂存区的文件提交到本地仓库，-m 后面是提交信息，必须清晰描述这次提交的内容。
    5.git remote add origin git@github.com:your_username/quadruped_ros2_control.git：添加远程仓库的关联，origin 是远程仓库的默认别名，后面是 SSH 地址（需要替换成自己的）。
    6.git push -u origin main：将本地 main 分支推送到 origin 远程的 main 分支，-u 是设置上游跟踪，后续 push 可以直接 git push。
    
    

          然后第二部分日常修改的流程：

    1.git pull origin main后修改代码：本地编辑代码文件。
    2.git add -A：再次把所有改动添加到暂存区，比git add . 覆盖范围更全。
    3.git status/git diff：git status 看哪些文件有改动；git diff 可以查看具体的代码改动内容（行级别的差异）。
    4.git commit -m "具体的改动说明"：提交到本地仓库，提交信息要具体（比如 “修复腿部控制的 PID 参数 bug”，而非 “修改代码”）。
    5.git pull origin main ：推送前再次拉取（双重保险，防止你修改代码的过程中有人推送新内容）
    6.git push：推送到远程仓库（因为之前用了 - u，所以这里不用指定分支和远程）。

	# ====================== 第一部分：第一次创建仓库并推送到 GitHub ======================
	# 初始化本地 Git 仓库，在当前文件夹创建隐藏的.git目录，标志该目录被Git版本控制接管
	git init  

	# 将本地默认分支从master重命名为main（GitHub当前默认分支为main，避免分支名不一致）
	# -m 是move/rename的简写，代表重命名分支
	git branch -m main  

	# 将当前目录下所有新增/修改的文件添加到Git暂存区（待提交缓冲区）
	# "." 代表当前目录及所有子目录，仅暂存新增/修改文件，不包含删除的文件
	git add .  

	# 将暂存区文件提交到本地Git仓库
	# -m 后是提交信息，fix: 是行业通用前缀（表示修复/调整），建议写具体内容（如"fix: 初始化个人网站基础页面"）
	git commit -m "fix: 第一次提交个人网站"  

	# 关联本地仓库到远程GitHub仓库
	# origin 是远程仓库的默认别名，后面替换为你自己的GitHub仓库HTTPS地址（需先在GitHub创建空仓库）
	git remote add origin https://github.com/Albert-Chen04/Video-Editing-Toolkit.git  

	# 强制将本地main分支推送到远程origin的main分支
	# -f 是force（强制）的简写，仅适合首次推送且远程仓库为空的场景！
	# 风险提示：若远程已有文件（如README），强制推送会覆盖远程内容，导致数据丢失
	# 更安全的替代方案：git push -u origin main（-u 绑定上游分支，后续推送只需git push）
	git push -f origin main  




	# ====================== 第二部分：本地修改代码后更新远程仓库 ======================
	#本地编辑代码文件。
	git pull origin main后修改代码
	
	# 全量暂存所有改动（新增/修改/删除的文件都包含），比git add . 覆盖范围更全
	# -A 是--all的简写，确保本地所有改动都被纳入暂存区
	git add -A  

	# 查看当前仓库文件状态，确认改动范围（已暂存文件显示绿色，未暂存显示红色）
	# 新手建议每次提交前执行，避免提交错误改动
	git status  

	# 提交本地第二次改动到Git仓库，提交信息需具体（如"fix: 修复导航栏点击失效问题"）
	git commit -m "fix: 第二次提交个人网站"  

	# 全局配置Git的HTTP代理，仅在无法科学上网、推送/拉取GitHub失败时使用
	# 127.0.0.1:7897 是VPN/代理工具的本地端口（不同工具端口不同，如Clash常用7890/7897）
	# 取消代理命令：git config --global --unset http.proxy
	git config --global http.proxy http://127.0.0.1:7897  

	# 拉取远程origin仓库main分支的最新代码到本地
	# 核心作用：避免多人协作/多设备修改导致的代码冲突，是日常更新的必做步骤
	git pull origin main  

	# 将本地提交的改动推送到远程main分支
	# 若首次推送用了git push -u origin main，此处可简化为git push
	git push origin main
